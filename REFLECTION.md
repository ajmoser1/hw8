The agent got pretty far given the relatively short length and gaps from the SPEC file. It was able to pass about 20/30 original criteria on the first build. I had to intervene in multiple problems. The first was that each question were created for the SPEC requirements. I originally asked for 9 questions but then changed it to 20 questions in the spec file but another part of the SPEC file mentioned having 9 questions, so it decided to go with less questions. The next problem was that even with 20 questions, I got repeat questions in the 6 randomly selected questions. I updated the question selection to exclude any repeats. 
The AI reviewer was certainly useful. It was able to audit the entire codebase and check against each criteria item accurately. It provided the right status for each item and made it simple to determine fixes.

To get a better result, I would undoubtedly put much more detail into the SPEC file about the questions and question selection. I should've structured it better as well, starting with the foundational features and format of the app and then moving into more complicated features. My spec file was a little scattered and I mixed in more complex features wherever.

Based on this experience, this is a great technique to prevent repeated debugging sessions from discovering bugs by using it myself. Although it doesn't catch everything, it's still fundamentally a great practice and can flag dozens of bugs. A good approach would be to run multiple reviewers, one focused on low-level details like structure, error handling, and so forth, and one for higher-level things like security and logins.
